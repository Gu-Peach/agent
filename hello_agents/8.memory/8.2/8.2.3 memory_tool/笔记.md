# 8.2.3 MemoryTool 记忆机制笔记

## 1. 总体结构

当前目录里的代码展示的是 HelloAgents 中 `MemoryTool` 的核心机制。

整体调用链路可以理解为：

```text
用户输入
  -> Agent / LLM 判断是否需要使用记忆工具
  -> 调用 MemoryTool
  -> MemoryTool 根据 action 分发 add / search / forget / consolidate 等操作
  -> MemoryManager 统一管理具体记忆类型
  -> WorkingMemory / EpisodicMemory / SemanticMemory / PerceptualMemory
  -> 底层存储与检索后端
```

在 Agent 层面，`MemoryTool` 是被注册进工具箱的工具：

```python
tool_registry.register_tool(memory_tool)
agent.tool_registry = tool_registry
```

所以 LLM 看到的不是 SQLite、Qdrant、Neo4j 这些数据库，而是一个统一的 `memory` 工具。  
数据库和具体记忆类型都被封装在 `MemoryTool` 和 `MemoryManager` 后面。

## 2. MemoryTool 的角色

`MemoryTool` 是 Agent 可以调用的记忆工具入口。

它支持的主要动作包括：

| action | 作用 |
|---|---|
| add | 添加记忆 |
| search | 检索记忆 |
| summary | 获取记忆摘要 |
| stats | 获取记忆统计 |
| update | 更新记忆 |
| remove | 删除记忆 |
| forget | 遗忘记忆 |
| consolidate | 整合记忆 |
| clear_all | 清空记忆 |

当前代码中的分发逻辑类似：

```python
def execute(self, action: str, **kwargs) -> str:
    if action == "add":
        return self._add_memory(**kwargs)
    elif action == "search":
        return self._search_memory(**kwargs)
    elif action == "summary":
        return self._get_summary(**kwargs)
```

也就是说，LLM 或 Agent 框架只需要给出：

```text
action + 对应参数
```

具体怎么存、怎么查，由工具内部完成。

## 3. MemoryTool 初始化

`MemoryTool` 初始化时会创建 `MemoryManager`。

```python
memory_tool = MemoryTool(
    user_id="user123",
    memory_types=["working", "episodic", "semantic"]
)
```

最基本参数：

| 参数 | 作用 | 默认值 |
|---|---|---|
| user_id | 当前用户 ID，用于隔离不同用户的记忆 | default_user |
| memory_config | 记忆系统配置 | MemoryConfig() |
| memory_types | 启用哪些记忆类型 | working、episodic、semantic |

如果启用：

```python
memory_types=["working"]
```

则只会使用工作记忆。

如果启用：

```python
memory_types=["working", "episodic", "semantic", "perceptual"]
```

则会启用全部记忆类型。

## 4. MemoryManager 的角色

`MemoryManager` 是具体记忆类型的统一管理器。

初始化时会根据开关创建对应模块：

```python
if enable_working:
    self.memory_types["working"] = WorkingMemory(...)

if enable_episodic:
    self.memory_types["episodic"] = EpisodicMemory(...)

if enable_semantic:
    self.memory_types["semantic"] = SemanticMemory(...)

if enable_perceptual:
    self.memory_types["perceptual"] = PerceptualMemory(...)
```

所以记忆类型的选择主要发生在 `MemoryManager`。

需要注意：

```text
LLM 不直接选择 WorkingMemory / EpisodicMemory / SemanticMemory 类。
LLM 通常只调用 MemoryTool，并传入 action、query、memory_type 等参数。
MemoryTool 再把请求交给 MemoryManager。
MemoryManager 根据 memory_type 或 memory_types 决定查哪些记忆模块。
```

在当前 `MemoryTool._add_memory()` 代码里：

```python
auto_classify=False
```

这表示添加记忆时不会让 `MemoryManager` 自动分类，而是使用调用方明确传入的 `memory_type`。

例如：

```python
memory_tool.execute(
    "add",
    content="Python 是一种解释型语言",
    memory_type="semantic",
    importance=0.9
)
```

这条记忆会被明确写入 `semantic` 类型，而不是由系统自动判断。

## 5. 添加记忆所需的最基本参数

添加记忆使用：

```python
memory_tool.execute("add", ...)
```

最小参数：

| 参数 | 是否必需 | 说明 |
|---|---|---|
| content | 基本必需 | 要保存的记忆内容 |
| memory_type | 可选 | 默认为 working |
| importance | 可选 | 重要性，默认 0.5 |
| file_path | perceptual 时可用 | 感知记忆的文件路径 |
| modality | perceptual 时可用 | text / image / audio 等 |
| metadata | 可选 | 其他额外字段，如 event_type、location、knowledge_type |

示例：

```python
memory_tool.execute(
    "add",
    content="用户张三正在学习 Agent 记忆系统",
    memory_type="episodic",
    importance=0.8
)
```

感知记忆示例：

```python
memory_tool.execute(
    "add",
    content="用户上传了一张 Python 代码截图",
    memory_type="perceptual",
    importance=0.7,
    modality="image",
    file_path="./uploads/code_screenshot.png"
)
```

## 6. 检索记忆所需的最基本参数

检索记忆使用：

```python
memory_tool.execute("search", ...)
```

最小参数：

| 参数 | 是否必需 | 说明 |
|---|---|---|
| query | 必需 | 检索问题 |
| limit | 可选 | 返回条数，默认 5 |
| memory_type | 可选 | 指定单个记忆类型 |
| memory_types | 可选 | 指定多个记忆类型 |
| min_importance | 可选 | 最小重要性阈值，默认 0.1 |

示例：

```python
memory_tool.execute(
    "search",
    query="用户最近在学习什么？",
    memory_type="episodic",
    limit=3
)
```

如果传入：

```python
memory_type="semantic"
```

内部会转换成：

```python
memory_types = ["semantic"]
```

如果不传 `memory_type` 或 `memory_types`，则通常会从当前启用的所有记忆类型中检索。

## 7. WorkingMemory：工作记忆

工作记忆用于保存当前会话中的短期信息。

特点：

```text
存储位置：内存
生命周期：短
访问速度：快
容量：有限
清理机制：TTL + 容量淘汰
```

添加链路：

```text
MemoryTool.add
  -> MemoryManager.add_memory
  -> WorkingMemory.add
  -> 先清理过期记忆
  -> 如果超过容量，删除低优先级记忆
  -> 写入内存列表
```

查询链路：

```text
MemoryTool.search
  -> MemoryManager.retrieve_memories
  -> WorkingMemory.retrieve
  -> 尝试 TF-IDF 检索
  -> 失败时回退关键词匹配
  -> 结合时间衰减和重要性排序
```

评分思路：

```text
final_score = base_relevance * time_decay * importance_weight
importance_weight = 0.8 + importance * 0.4
```

适合保存：

```text
用户刚刚问了什么
当前任务上下文
临时推理状态
本轮对话短期信息
```

## 8. EpisodicMemory：情景记忆

情景记忆用于保存具体事件和经历。

特点：

```text
存储内容：事件、经历、对话片段、用户行为
主要后端：SQLite + Qdrant
检索方式：结构化过滤 + 向量语义检索
```

添加链路：

```text
MemoryTool.add
  -> MemoryManager.add_memory
  -> EpisodicMemory.add
  -> 创建 Episode 对象
  -> 更新 session_id -> episode_id 索引
  -> 写入 SQLite
  -> 生成 embedding
  -> 写入 Qdrant
```

SQLite 中保存完整结构化记录：

```text
memory_id
user_id
content
memory_type
timestamp
importance
metadata / context
session_id
```

Qdrant 中保存：

```text
id = memory_id
vector = content embedding
payload = user_id、memory_type、importance、session_id、content 等
```

查询链路：

```text
MemoryTool.search
  -> MemoryManager.retrieve_memories
  -> EpisodicMemory.retrieve
  -> SQLite 做时间、重要性、用户等结构化过滤
  -> Qdrant 做语义相似度检索
  -> 合并候选结果
  -> 按向量相似度、时间近因性、重要性排序
```

评分思路：

```text
base_relevance = vector_score * 0.8 + recency_score * 0.2
importance_weight = 0.8 + importance * 0.4
final_score = base_relevance * importance_weight
```

适合保存：

```text
某天发生的事
用户完成了某个项目
用户曾经表达过某个偏好
一次具体对话或行为记录
```

## 9. SemanticMemory：语义记忆

语义记忆用于保存抽象知识、事实、概念和关系。

特点：

```text
存储内容：知识、事实、概念、规则、实体关系
主要后端：Qdrant + Neo4j
检索方式：向量搜索 + 图搜索
```

添加链路：

```text
MemoryTool.add
  -> MemoryManager.add_memory
  -> SemanticMemory.add
  -> 对 content 生成 embedding
  -> 抽取实体 entities
  -> 抽取关系 relations
  -> 实体和关系写入 Neo4j
  -> embedding 和 payload 写入 Qdrant
```

Qdrant 保存语义向量：

```text
id = memory_id
vector = content embedding
payload = memory_id、entities、entity_count、relation_count 等
```

Neo4j 保存图结构：

```text
(Entity)-[RELATION]->(Entity)
```

当前实现里，图结构更接近基础图谱：

```text
实体来自 NLP / spaCy 抽取
关系主要是共现关系 CO_OCCURS
也可能保存词法依存关系
```

查询链路：

```text
MemoryTool.search
  -> MemoryManager.retrieve_memories
  -> SemanticMemory.retrieve
  -> Qdrant 做向量检索
  -> Neo4j 做图关系检索
  -> 合并 vector_results 和 graph_results
  -> 计算综合得分
  -> 返回排序后的 MemoryItem
```

评分思路：

```text
base_relevance = vector_score * 0.7 + graph_score * 0.3
importance_weight = 0.8 + importance * 0.4
combined_score = base_relevance * importance_weight
```

适合保存：

```text
Python 是解释型语言
张三是 Python 开发者
推荐系统项目使用 Qdrant 和 Neo4j
某个概念、规则、知识点
```

## 10. PerceptualMemory：感知记忆

感知记忆用于保存文本、图片、音频等多模态信息。

特点：

```text
存储内容：文本、图像、音频等感知数据
主要后端：按模态分离的 Qdrant collection
检索方式：同模态或跨模态向量检索
```

初始化时会维护多个向量库：

```python
self.vector_stores = {
    "text": perceptual_text,
    "image": perceptual_image,
    "audio": perceptual_audio,
}
```

添加时最重要的参数是：

```text
content
memory_type="perceptual"
file_path
modality
```

查询链路：

```text
MemoryTool.search
  -> MemoryManager.retrieve_memories
  -> PerceptualMemory.retrieve
  -> 根据 query_modality 编码 query
  -> 根据 target_modality 选择对应向量库
  -> Qdrant 相似度检索
  -> 结合时间近因性和重要性排序
```

评分思路：

```text
base_relevance = vector_score * 0.8 + recency_score * 0.2
importance_weight = 0.8 + importance * 0.4
combined_score = base_relevance * importance_weight
```

适合保存：

```text
用户上传的代码截图
音频片段
图片内容
多模态文件及其描述
```

## 11. 遗忘机制

遗忘通过：

```python
memory_tool.execute("forget", ...)
```

最基本参数：

| 参数 | 说明 |
|---|---|
| strategy | 遗忘策略 |
| threshold | 重要性阈值 |
| max_age_days | 最大保留天数 |

支持三类策略：

| strategy | 作用 |
|---|---|
| importance_based | 删除重要性低于阈值的记忆 |
| time_based | 删除超过指定天数的记忆 |
| capacity_based | 当容量超限时删除较不重要的记忆 |

示例：

```python
memory_tool.execute(
    "forget",
    strategy="importance_based",
    threshold=0.2
)
```

## 12. 整合机制

整合通过：

```python
memory_tool.execute("consolidate", ...)
```

最基本参数：

| 参数 | 说明 |
|---|---|
| from_type | 来源记忆类型 |
| to_type | 目标记忆类型 |
| importance_threshold | 整合阈值 |

示例：

```python
memory_tool.execute(
    "consolidate",
    from_type="working",
    to_type="episodic",
    importance_threshold=0.7
)
```

含义是：

```text
把重要性 >= 0.7 的工作记忆提升为情景记忆。
```

也可以进一步：

```python
memory_tool.execute(
    "consolidate",
    from_type="episodic",
    to_type="semantic",
    importance_threshold=0.8
)
```

含义是：

```text
把重要的情景记忆抽象为语义记忆。
```

## 13. LLM、MemoryTool、MemoryManager 的分工

一个容易混淆的点是：

```text
LLM 是否通过 MemoryManager 来选择记忆的存储和查询？
```

更准确的说法是：

```text
LLM 不直接调用 MemoryManager。
LLM 调用的是注册到工具箱中的 MemoryTool。
MemoryTool 内部再调用 MemoryManager。
MemoryManager 根据 memory_type / memory_types 分发到具体记忆模块。
```

也就是：

```text
LLM
  -> MemoryTool
  -> MemoryManager
  -> WorkingMemory / EpisodicMemory / SemanticMemory / PerceptualMemory
  -> SQLite / Qdrant / Neo4j / 内存
```

### 添加记忆时

当前 `MemoryTool._add_memory()` 明确传入：

```python
auto_classify=False
```

因此添加记忆时，主要由调用参数里的 `memory_type` 决定存到哪种记忆。

```text
memory_type="working"    -> WorkingMemory
memory_type="episodic"   -> EpisodicMemory
memory_type="semantic"   -> SemanticMemory
memory_type="perceptual" -> PerceptualMemory
```

### 检索记忆时

检索时如果指定：

```python
memory_type="episodic"
```

则只查情景记忆。

如果指定：

```python
memory_types=["working", "episodic"]
```

则查多个记忆类型。

如果不指定，通常会查当前启用的全部记忆类型。

## 14. 当前机制的一句话总结

当前 `8.2.3 memory_tool` 的记忆机制可以概括为：

```text
MemoryTool 是 Agent 暴露给 LLM 的统一记忆工具；
MemoryManager 是内部调度中心；
不同 memory_type 对应不同记忆模块；
不同记忆模块使用不同存储和检索后端；
LLM 只需要提出 action/query/content/memory_type 等参数，不需要直接操作数据库。
```
