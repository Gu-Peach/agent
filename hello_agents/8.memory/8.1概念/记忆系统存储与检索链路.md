# 记忆系统的存储形式与检索链路

## 1. 核心理解

当前 memory 系统可以理解为一个增强版 RAG 记忆系统。

普通 RAG 主要做：

```text
文本 -> Embedding -> 向量数据库 -> 相似度检索 -> 返回相关文本
```

memory 系统在这个基础上增加了记忆类型、重要性、时间、用户、会话、实体关系等信息。因此它不只是“找相似文档”，而是把一次对话、一条事实、一个事件，都包装成 `MemoryItem` 再进行存储和检索。

基础记忆结构类似：

```python
MemoryItem(
    id="uuid",
    content="用户张三是一名 Python 开发者",
    memory_type="semantic",
    user_id="user123",
    timestamp=datetime.now(),
    importance=0.8,
    metadata={
        "session_id": "...",
        "entities": [...],
        "tags": [...]
    }
)
```

## 2. 记忆类型

系统里主要有几类记忆：

| 类型 | 含义 | 典型内容 | 主要存储 |
|---|---|---|---|
| working | 工作记忆 | 当前短期上下文、最近对话 | 内存 |
| episodic | 情景记忆 | 某次对话、某个事件、某个用户行为 | SQLite + Qdrant |
| semantic | 语义记忆 | 稳定事实、知识、人物画像、概念关系 | Qdrant + Neo4j |
| perceptual | 感知记忆 | 图片、音频、文件等多模态信息 | SQLite + Qdrant |

简单聊天记忆通常只需要 `working` 和 `episodic`。  
如果要做长期知识积累、人物关系、概念网络、多跳推理，才需要 `semantic` 和 Neo4j。

## 3. 各数据库的职责

### 3.1 SQLite：权威文本存储

SQLite 更像是“原始记忆档案库”。

它保存完整、结构化、可恢复的记忆记录，例如：

```text
memory_id
user_id
content
memory_type
timestamp
importance
properties / metadata
```

适合做：

- 按用户过滤
- 按时间过滤
- 按记忆类型过滤
- 按重要性过滤
- 保存完整原文
- 作为 Qdrant 检索结果的详情来源

例如一条 episodic memory：

```json
{
  "memory_id": "m001",
  "user_id": "user123",
  "content": "用户张三今天说他正在学习 Python Agent 记忆系统",
  "memory_type": "episodic",
  "timestamp": 1760000000,
  "importance": 0.8,
  "properties": {
    "session_id": "session_20260613",
    "tags": ["python", "agent", "memory"]
  }
}
```

### 3.2 Qdrant：向量相似度检索

Qdrant 是向量数据库，负责回答：

```text
当前问题和哪些记忆语义最相似？
```

存储形式不是直接只存文本，而是：

```json
{
  "id": "m001",
  "vector": [0.012, -0.233, ...],
  "payload": {
    "memory_id": "m001",
    "user_id": "user123",
    "memory_type": "episodic",
    "importance": 0.8,
    "session_id": "session_20260613",
    "content": "用户张三今天说他正在学习 Python Agent 记忆系统"
  }
}
```

其中：

- `vector` 是 content 经过 embedding 模型得到的向量
- `payload` 是过滤和回填信息
- `memory_id` 用来和 SQLite / Neo4j 里的记录关联

适合做：

- 语义搜索
- 模糊匹配
- 相似记忆召回
- RAG 上下文构建

例如用户问：

```text
我最近在学什么？
```

即使原文没有“最近在学什么”这几个字，Qdrant 也能通过语义相似度找回：

```text
用户张三今天说他正在学习 Python Agent 记忆系统
```

### 3.3 Neo4j：实体关系与图推理

Neo4j 是图数据库，负责回答：

```text
记忆里的实体之间有什么关系？
```

它不是为了替代 Qdrant，而是补充 Qdrant 不擅长的“关系链查询”。

例如文本：

```text
张三是 Python 开发者，参与了推荐系统项目。推荐系统项目使用 Qdrant 和 Neo4j。
```

可以抽取成图结构：

```text
(张三)-[:IS_A]->(Python开发者)
(张三)-[:PARTICIPATES_IN]->(推荐系统项目)
(推荐系统项目)-[:USES]->(Qdrant)
(推荐系统项目)-[:USES]->(Neo4j)
```

Neo4j 中节点可能类似：

```json
{
  "entity_id": "person_zhangsan",
  "name": "张三",
  "entity_type": "PERSON",
  "memory_id": "m001",
  "user_id": "user123"
}
```

关系可能类似：

```json
{
  "from_entity": "person_zhangsan",
  "to_entity": "project_recommendation",
  "relation_type": "PARTICIPATES_IN",
  "memory_id": "m001",
  "evidence": "张三参与了推荐系统项目"
}
```

适合做：

- 人物关系查询
- 项目关系查询
- 技术栈关联查询
- 多跳推理
- 知识图谱构建

例如用户问：

```text
张三参与的项目用了哪些数据库？
```

这个问题不是简单相似度问题，而是关系链问题：

```text
张三 -> 参与 -> 推荐系统项目 -> 使用 -> Qdrant / Neo4j
```

这种检索就是图数据库的优势。

## 4. 一条记忆的写入链路

以这条记忆为例：

```text
用户张三是一名 Python 开发者，正在学习 Agent 记忆系统。
```

### 4.1 包装成 MemoryItem

```text
content: 用户张三是一名 Python 开发者，正在学习 Agent 记忆系统。
memory_type: semantic 或 episodic
user_id: user123
importance: 0.8
metadata: session_id、tags、source 等
```

### 4.2 写入 SQLite

如果是 episodic / perceptual 这类需要原始档案的记忆，会写入 SQLite：

```text
memory_id -> 原文、用户、时间、类型、重要性、metadata
```

SQLite 是后续“取详情”的可靠来源。

### 4.3 写入 Qdrant

系统调用 embedding 模型：

```text
content -> embedding vector
```

然后写入 Qdrant：

```text
id: memory_id
vector: embedding
payload: user_id、memory_type、importance、content、session_id
```

之后就可以通过语义相似度召回。

### 4.4 写入 Neo4j

如果启用了 semantic memory，系统会尝试做实体抽取：

```text
张三
Python 开发者
Agent 记忆系统
```

并建立关系：

```text
张三 -> IS_A -> Python开发者
张三 -> LEARNS -> Agent记忆系统
```

这些结构会写入 Neo4j，用于关系查询。

## 5. 查询链路

### 5.1 普通语义问题

问题：

```text
张三在学什么？
```

检索链路：

```text
用户问题
  -> 转成 query embedding
  -> 到 Qdrant 做相似度搜索
  -> 找到相关 memory_id
  -> 根据 memory_id 读取 payload 或 SQLite 原文
  -> 按相似度、重要性、时间排序
  -> 返回给 Agent 作为上下文
```

适合 Qdrant。

### 5.2 时间/重要性过滤问题

问题：

```text
最近比较重要的用户偏好是什么？
```

检索链路：

```text
用户问题
  -> SQLite 先按 user_id、time_range、importance 过滤候选
  -> Qdrant 做语义相似度召回
  -> 两边结果按 memory_id 合并
  -> 综合 vector_score、recency_score、importance
  -> 返回记忆
```

这里 SQLite 负责结构化过滤，Qdrant 负责语义匹配。

### 5.3 关系推理问题

问题：

```text
张三参与的项目用了哪些数据库？
```

检索链路：

```text
用户问题
  -> 抽取实体：张三、项目、数据库
  -> 到 Neo4j 查找张三相关实体
  -> 沿关系边多跳遍历
  -> 找到 项目 -> 使用 -> 数据库
  -> 得到相关 memory_id
  -> 回查 Qdrant / SQLite 获取原始记忆文本
  -> 返回给 Agent
```

适合 Neo4j。

### 5.4 混合检索问题

问题：

```text
张三最近在推荐系统项目里主要接触了哪些技术？
```

这个问题同时需要：

- “张三”
- “推荐系统项目”
- “最近”
- “技术”
- 语义相似
- 实体关系

混合链路：

```text
用户问题
  -> Qdrant 找语义相似记忆
  -> SQLite 根据时间、用户、重要性过滤
  -> Neo4j 查实体关系链
  -> 合并结果
  -> 去重
  -> 排序
  -> 生成上下文
```

这就是 memory 系统比普通 RAG 更复杂的地方。

## 6. 三类存储的对比

| 存储 | 存什么 | 擅长什么 | 不擅长什么 |
|---|---|---|---|
| SQLite | 原文、时间、用户、类型、重要性、metadata | 精确过滤、持久化、取详情 | 语义相似度 |
| Qdrant | embedding 向量 + payload | 语义搜索、模糊召回 | 复杂关系推理 |
| Neo4j | 实体节点 + 关系边 | 多跳关系、知识图谱、关联推理 | 模糊语义搜索 |

三者一起使用时，可以形成：

```text
SQLite = 记忆档案库
Qdrant = 语义检索引擎
Neo4j = 关系推理地图
```

## 7. 为什么不是只用 Qdrant？

只用 Qdrant 可以解决很多问题，但它主要靠“相似度”。

例如：

```text
张三参与的项目用了哪些数据库？
```

如果原始文本里没有一句非常接近这个问题，向量检索可能召回不稳定。

Neo4j 可以把问题拆成确定关系：

```text
张三 -> 项目 -> 数据库
```

所以：

- 简单记忆：Qdrant 足够
- 长期知识：SQLite + Qdrant 更稳
- 关系推理：需要 Neo4j

## 8. 当前项目里的现实建议

在你当前复现阶段，可以先只启用：

```python
MemoryTool(user_id="user123", memory_types=["working", "episodic"])
```

这样可以先跑通：

```text
内存短期记忆 + SQLite 持久化 + Qdrant 语义检索
```

等 Neo4j Aura 网络稳定后，再启用：

```python
MemoryTool(user_id="user123", memory_types=["working", "episodic", "semantic"])
```

这样才会加入：

```text
实体抽取 + 图数据库 + 关系推理
```
